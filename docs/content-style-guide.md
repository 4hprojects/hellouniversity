# HelloUniversity Content Style Guide

Updated: 2026-03-30

## Core Rule

All public-facing and in-app copy should be written from the user's point of view, not the developer's point of view.

The default question is:

"What would a normal visitor, student, teacher, or admin naturally need to understand here?"

Copy should explain what users can do, what they can see, and why it is useful. It should not expose internal system language unless the audience truly needs it.

## Writing Standard

- Prefer plain language over internal product language.
- Describe actions and outcomes, not implementation details.
- Write for visitors, students, teachers, admins, and reviewers who are not reading the codebase.
- Use terms that a normal user would say out loud.
- If a phrase sounds like architecture, product planning, or developer shorthand, rewrite it.

## Default User Questions

Most product copy should help answer one or more of these:

- What can I do here?
- What can I access before login?
- What changes after login?
- What is this page for?
- Why is this useful to me?
- What should I do next?

## Avoid Developer POV

Avoid terms like these in user-facing copy unless there is a strong reason:

- role-aware
- role-based
- workflow
- analytics views
- response flow
- grade-related views
- internal linking
- discoverable pages
- product explanation
- implementation details
- canonical page
- system layer
- route-aware
- first-class workflow

## Prefer User POV

Prefer wording like:

- teachers can manage classes
- students can check attendance and grades
- visitors can browse lessons before login
- users can find help and support pages easily
- teachers can assign quizzes to a class
- students can open activities and see what is due
- this page helps you understand what the platform offers

## Rewrite Examples

- "role-aware experience" -> "different views for teachers and students"
- "student response flow" -> "students can answer quizzes online"
- "analytics views" -> "teachers can review quiz results"
- "grade-related views" -> "students can check grades"
- "internal linking and discoverable pages" -> "lessons, guides, and help pages are easier to find"
- "product explanation" -> "clear information about what the platform does"

## Scope

Apply this guideline to:

- public marketing and product pages
- help and support pages
- blog intros and platform explainers
- dashboard helper text
- empty states
- onboarding and approval-related copy
- buttons, labels, and section headings where plain language is possible

## Exception

Technical or admin-only documentation can use more precise internal terms when accuracy matters more than simplicity. Even then, prefer the clearest wording available.
